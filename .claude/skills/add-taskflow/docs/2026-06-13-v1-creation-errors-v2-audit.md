## Residual hole the RC1 adversarial verifier found (genuine, NEW — not the v1 bug)

`pickUniqueAgentFolder` dedups the new folder **only against `agent_groups.folder`, never against `boards.group_folder`** (`provision-shared.ts:616`). The 9 migrated boards carry drifted `boards.group_folder` values (e.g. `board-geo-secti-taskflow` whose `group_folder='anali-geo-taskflow'`) that by definition don't appear in `agent_groups.folder`. `boards.group_folder` has no UNIQUE constraint, and `resolveTaskflowBoardId`'s direct lookup is `… WHERE group_folder=? LIMIT 1` with **no ORDER BY** (`taskflow-db.ts:854`). So a new board provisioned with an operator/model-supplied `group_folder` override that collides with a drifted value inserts a *second* board sharing that `group_folder`, violating the cutover invariant `GROUP BY group_folder HAVING count>1 is empty` (`docs/v2-cutover-exception-list.md:42`) and making the migrated board's wake resolution nondeterministic. Suggested v2.1 hardening: dedup the new folder against `boards.group_folder` too, or add a UNIQUE/preflight on it.

- **RC5 — escalated PARTIAL → NOT_FIXED.** Both verifiers independently confirmed the hole AND found it worse: root-board path is affected too, no post-create rescue, silent CREATE-time person-drop. This is the top cutover risk.

## Still open at cutover (prioritized)

1. **RC5 9th-digit reconciliation — FIXED 2026-06-13** (board-people + provisioning surface). Create-time: both root and child provision (and the standalone `create_group`) now resolve the participant JID via WhatsApp's `onWhatsApp()` round-trip (`resolveParticipantJid`), so the server-canonical 9th-digit form is used instead of a silently-dropped string-built JID. Runtime: `actor-person-resolution` and the cross-parent dedup (`findExistingBoardElsewhere`) match against a `brPhoneMatchVariants` 12↔13 equivalence set (conservative to the 2012-migration `9[6-9]` shape so landlines can't be mis-matched), keeping the exactly-one-match fail-closed guard. Two Codex gpt-5.5/xhigh rounds: R1 found a real mis-attribution BLOCKER (landline variant) + 2 IMPORTANTs (cross-parent exact-match duplicate; `create_group` string-concat) — all fixed; R2 confirmed correct. **RC5-ext (external_contacts) — phone-match sites FIXED 2026-06-13:** all `external_contacts.phone` lookups now resolve across the BR 9th-digit split via a shared `findExternalContactByPhone` helper (exact match first → single `brPhoneMatchVariants` match → undefined on ambiguity, fail-closed): `add_external_participant` (dedup — no duplicate contact when re-added with the other form), `remove_external_participant` (a revoke by the twin form now finds the contact instead of leaving the grant active), and `reinvite_external_participant`. `resolveExternalDm` (`src/dm-routing.ts`) likewise matches the variant set + backfills `direct_chat_jid` — **but note it is currently UNWIRED** (no call site; external inbound-DM resolution is documented as not-yet-wired in `taskflow-helpers.ts:206`), so that fix is forward-looking/latent until the resolver is wired. Codex gpt-5.5/xhigh: no BLOCKER; confirmed the variant logic is landline-safe and fail-closed.

**Residual (narrow, documented) — RC5-ext site-3, re-verified 2026-06-13 (assessment, no code change warranted):** the engine's notification fallback `target_chat_jid: ext.direct_chat_jid ?? \`${ext.phone}@s.whatsapp.net\`` (`taskflow-engine.ts:4088`) still string-builds the JID for a never-contacted external. **Verified safe:** the container has no `onWhatsApp()` socket to round-trip, and host delivery `deliverTextToWhatsAppJid` (`taskflow-notify.ts:47-51`) **fail-closes** — `getMessagingGroupByPlatform('whatsapp', jid)` returns nothing for a never-contacted external (no DM messaging_group exists, which is exactly why `direct_chat_jid` is null), so it `log.error`s and returns false. This is a **non-delivery, already fail-loud-logged — never a mis-delivery** (a collision would require the wrong 9th-digit form to exactly equal a *different* contact's real JID, a degenerate phone-data coincidence). No bounded fix exists at this layer: canonicalizing the phone here changes nothing (still no messaging_group → still non-delivery). Actually delivering the first-ever outbound to a never-contacted external needs the external inbound-DM resolver wired + a host-side `onWhatsApp()` round-trip at delivery + cold-DM `messaging_group` provisioning (touches all direct_messages) — a separate, larger unit, deferred by design.
2. **RC1 folder-collision — FIXED 2026-06-13.** Provision now dedups the new folder against **board** folders too: `getReservedBoardFolders(TASKFLOW_DB_PATH)` (taskflow-db.ts; `boards.group_folder` ∪ `board_groups.group_folder`, incl. migrated-drifted values, fail-soft) is unioned with `agent_groups.folder` at both root + child provision, so a new board can't be created with a `group_folder` colliding with a migrated board's drifted value. Belt-and-suspenders: the direct `boards WHERE group_folder=?` lookups in `resolveTaskflowBoardId` + `resolveBoardTimezone` now `ORDER BY id`, so resolution stays deterministic even if a collision ever existed. TDD'd (getReservedBoardFolders unit + provision suites green).
3. **RC3 silent partial — FIXED 2026-06-13** (the dangerous one). The unresolvable-parent path (`if (!parent) return`) was **fully silent** — register_person had already committed the person, so it left a registered person with no board and no trace. Now `log.error` (fail-loud/diagnosable). A chat alert isn't possible there because the parent IS the alert's destination (`sourceMessagingGroupPlatformId`) — exactly what failed to resolve. The other early returns are either already `log.warn` (`!parsed`, `!adapter`) or an intentional dedup-success (`alreadyOnThisParent`); the dangerous post-seed failure paths already `alertProvisionFailed` (EX-014).
4. **RC6 anti-abuse rate-limit — operational, not a code fix.** A WhatsApp anti-abuse logout after ~6 rapid group creations is a one-time *cutover-day* concern (bulk board import). A permanent runtime rate-limiter would over-engineer a one-off; the cutover runbook's "pace bulk provisioning (don't fire >5 rapidly)" is the right control. Left as a runbook item by design.
5. **RC4 concurrent-cross-session orphan group — FIXED 2026-06-13.** A truly-parallel double-provision of the same `(parent, person)` from two different host sessions could both pass the non-transactional `alreadyOnThisParent` SELECT, both call `adapter.createGroup` (= two WhatsApp groups), then the 2nd registration INSERT roll back — leaving an orphan group. Closed with a **dedicated lock table** `child_board_provision_claims` (PK `(parent_board_id, person_id)`, read by nothing else so `child_board_registrations` keeps its "a row ⇒ a seeded board" invariant): pre-claim the lock BEFORE `createGroup` (PK conflict → concurrent loser skips the group side-effect); **double-checked locking** re-reads `child_board_registrations` under the lock before `createGroup` (closes the slow-contender TOCTOU where a session passes the first SELECT before the winner seeds, then acquires the lock after the winner releases); registration INSERT stays inside `seedChildTaskflow`'s transaction; the lock is released on success and on every failure path. `isPrimaryKeyConflict` matches ONLY `SQLITE_CONSTRAINT_PRIMARYKEY`/`UNIQUE` (FK/NOT NULL propagate as throws). **FOUR Codex gpt-5.5/xhigh rounds**: R1 found 2 BLOCKERs (dangling-claim recovery deleted active claims; releaseClaim not ownership-scoped) + 2 IMPORTANTs; R2 found the consumer-ripple from transient NULL rows → drove the dedicated-table redesign; R3 found the slow-contender TOCTOU → double-checked locking; R4 CLEAN for the same-`(parent, person)` race. Residual (narrow operational): a lock orphaned by a hard process crash in the (claim → seed) window blocks re-provisioning of that one person until cleared (no auto-takeover, since a lock row can't be distinguished from an active one without a token; `claimed_at` aids diagnosis).

   **Newly identified (Codex R4, separate from RC4, deferred):** two DIFFERENT parents provisioning the SAME person/phone concurrently both miss `findExistingBoardElsewhere` (the lock is `(parent, person)`-scoped) → two fully-seeded child boards instead of one linked board. This is NOT an orphan group (both boards are valid and post-hoc linkable/mergeable) — it violates the cross-parent reuse model under a narrow cross-parent timing window. Closing it needs a person/phone-scoped (canonicalized for the BR 9th-digit split) claim taken BEFORE `findExistingBoardElsewhere` — a change to the cross-parent unification model, out of RC4's scope.

   RC5-ext site-3 notification JID-build (above) is the remaining external-contacts residual.

**Net:** the two errors that caused the visible v1 *failure storms* — the resolution-key retry loop (RC1) and the duplicate-board multiplication (RC4) — are genuinely fixed, and folder-rename routing (RC2) can no longer happen. The remaining exposure is phone-matching (RC5) and cutover-pace anti-abuse (RC6), plus two deferred-to-v2.1 residuals (RC1 folder-collision, RC3 silent partials).
